# Localization and Internationalization Rules

## Purpose

ScreenPlay Pro must support future localization from the beginning.

The app should be usable by writers around the world without rewriting the codebase.

## Core Rule

Never hardcode user-facing strings in components, pages, buttons, forms, labels, modals, toast messages, tooltips, onboarding screens, empty states, navigation, or error messages.

All user-facing strings must use translation keys.

## Language Types

ScreenPlay Pro separates:

1. UI Language — the language of the app interface.
2. Project Language — the language the project is written in.
3. Screenplay Language — the primary language of the script text.
4. Academy Language — the language used to teach the user.
5. AI Response Language — the language the AI uses when helping.
6. Export Language — the language used for pitch decks, reports, and exports.
7. Collaborator Language — a collaborator's personal interface language.

Do not assume these are the same.

## Example

A user may:

- use the UI in Polish
- write a screenplay in English
- receive Academy instruction in Ukrainian
- export a pitch deck in English

## Bad

```tsx
<button>Save Project</button>
```

## Good

```tsx
<button>{t('project.save')}</button>
```

## Error Handling

Errors must be localized.

Bad:

```ts
toast.error('Something went wrong')
```

Good:

```ts
toast.error(t('errors.generic'))
```

## Empty States

Empty states must be localized too.

Bad:

```tsx
<p>No projects yet.</p>
```

Good:

```tsx
<p>{t('emptyStates.noProjects')}</p>
```

## AI Response Language

Internal AI system prompts may be written in English.

User-facing AI responses should respect the user's selected AI response language.

If the project language and UI language differ, the user should be able to choose which language AI uses for explanations.

## Database Preferences

Store user-level language preferences separately:

```text
profiles.ui_language
profiles.preferred_ai_language
profiles.academy_language
```

Store project-level language preferences separately:

```text
projects.project_language
projects.screenplay_language
projects.default_export_language
```

## Developer Rule

When adding a new page or feature, add translation keys at the same time.

Do not postpone localization.

Do not merge new UI with hardcoded user-facing copy unless it is a temporary developer-only debug label and clearly marked for removal.

## Stage 1 Requirement

For Stage 1, the app does not need complete translations into multiple languages yet.

It does need the architecture to avoid hardcoded strings so translation files can be expanded later.
